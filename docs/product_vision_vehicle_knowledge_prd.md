# 摩托车售后管理与维修知识系统产品设想

## 1. 产品定位

本项目不是一个普通的门店后台，也不是只记录客户、车辆和工单的业务系统。

本项目的长期目标，是建设一套面向摩托车售后维修门店的“经营管理系统 + 车型标准库 + 维修知识系统”一体化平台。

系统既要满足门店日常高频经营使用，也要逐步沉淀门店自己的专业知识资产，让门店从“能接车、能开单、能交车”升级为“有标准、有证据、有专业壁垒”的成熟维修服务体系。

一句话概括：

“建设一套以车辆和车型知识为核心的摩托车售后管理系统，在实现客户、车辆、工单、报价、配件、交付全流程闭环的同时，将车型维修手册、标准工艺、扭矩参数、作业方法和检查标准逐步数字化，形成技师可执行、门店可管理、客户可感知的专业维修知识资产。”

## 2. 核心产品思路

### 2.1 不只是业务流转，而是业务流转 + 技术知识沉淀

传统门店系统通常只关注：

- 客户是谁
- 车是什么
- 做了什么项目
- 收了多少钱

但本项目希望进一步解决：

- 这台车应该怎么修
- 这个项目按什么标准做
- 使用了什么配件和油液
- 关键扭矩是多少
- 哪些检查点必须复核
- 交付给客户时，如何证明门店是专业、严谨、按标准施工的

### 2.2 以“车辆”为经营和知识的核心单位

系统最终不是以“客户”作为唯一核心，而是以“车辆”作为核心资产单位。

每一辆车应长期沉淀以下内容：

- 基础档案
- 客户归属
- 历史工单
- 历史报价
- 历史体检记录
- 配件消耗记录
- 标准保养项目
- 标准维修项目
- 维修手册与资料库
- 数字化维修知识

这样一辆车在系统里，不再只是一次交易对象，而是一份持续积累的长期技术档案。

## 3. 产品能力分层

### 3.1 第一层：业务运营层

这一层解决“门店能不能顺畅经营”的问题。

应包含：

- 客户建档与客户管理
- 客户名下车辆管理
- 快速接车
- 工单流转
- 报价与版本管理
- 配件与库存记录
- 整车体检与状态跟踪
- 交付确认
- 单据打印与归档

目标是让门店从接车到交付结算形成清晰、稳定、可追溯的业务闭环。

### 3.2 第二层：车型标准库

这一层解决“同类车型能不能标准化服务”的问题。

每个车型应当不只是一个名称，而是一套完整的标准资料载体。车型下面应长期维护：

- 品牌
- 车型
- 年份范围
- 排量
- 分类
- 发动机代号
- 标准保养项目
- 标准维修项目
- 建议工时
- 建议价格
- 所需配件
- 保养周期
- 常见注意事项

当门店处理某一台具体车辆时，工单应优先从“车型标准库”里直接复用，而不是每次临时手工输入。

### 3.3 第三层：维修知识层

这一层解决“技师有没有可执行的标准作业依据”的问题。

知识层由两部分组成：

#### 3.3.1 资料库

资料库负责保存原始资料，主要包括：

- 原厂维修手册 PDF
- 扭矩表
- 线路图
- 保养标准
- 故障诊断资料
- 技术通告

资料库按车型关联，技师应能按车型快速定位和打开相关资料。

#### 3.3.2 数字化维修手册

数字化维修手册负责把原始 PDF 中真正有价值的技术信息结构化沉淀出来，包括：

- 标准维修条目
- 标准作业步骤
- 所需工具
- 扭矩数据
- 油液规格
- 间隙或限值
- 关键检查点
- 风险提示
- 客户可展示的专业说明

这部分不是简单的文件存储，而是将维修知识从“文档”变成“系统可执行的数据”。

### 3.4 第四层：专业证明层

这一层解决“如何向客户证明本次施工专业、规范、严谨”的问题。

未来客户侧或打印单据中，除了显示“换了什么”，还应逐步呈现：

- 本次维修保养项目
- 所用配件与油液
- 执行的标准检查项
- 关键施工参数
- 是否按标准扭矩复紧
- 完工体检记录
- 关键风险项是否已复核
- 技师签字

目标不是把客户单据做得复杂，而是让客户能够感知门店的专业标准，而不是只看到简单的收费结果。

## 4. 三个核心对象

### 4.1 车辆

车辆是系统中的长期资产单位。

围绕每一辆车，应形成完整的生命周期数据：

- 基础档案
- 客户归属
- 历次进店记录
- 工单历史
- 报价历史
- 体检时间线
- 维修项目历史
- 配件使用历史
- 技术资料和适用标准

### 4.2 维修项目

维修项目不能只是一个文本名称，而应当成为标准作业单元。

每个维修项目未来至少应包含：

- 项目名称
- 项目编码
- 适用车型
- 工时
- 建议售价
- 所需配件
- 维修方法
- 标准步骤
- 扭矩与关键参数
- 质检点
- 对客展示说明

### 4.3 资料与知识

资料和知识应区分层级：

- 原始资料：原厂 PDF、图片、图表
- 解析草稿：自动识别出来但未确认的结构化内容
- 已审核知识：人工确认后的可用知识
- 已发布标准：正式可用于工单、技师作业和客户展示的标准内容

这套层级可以保证知识沉淀既高效又稳妥，不会因为一次错误识别直接污染正式标准库。

## 5. 未来业务工作流设想

### 5.1 前台接车

- 选择老客户或快速新建客户
- 在客户名下选择车辆或新增车辆
- 填写客户主诉
- 生成工单

### 5.2 工单进入施工前

- 系统根据该车对应车型，自动加载标准维修项目
- 店员或技师从车型标准项目中直接勾选本次施工项目
- 系统带出建议工时、建议价格、所需配件
- 生成报价版本

### 5.3 施工过程中

- 技师在工单中查看项目说明
- 如该项目已关联数字化维修手册，则可直接查看：
  - 标准步骤
  - 所需工具
  - 扭矩数据
  - 注意事项
  - 关键检查点
- 如需查看更多原始资料，可一键打开该车型的 PDF 资料库

### 5.4 完工后

- 补录施工过程记录
- 做整车体检
- 记录最终数据
- 准备交付

### 5.5 交付给客户

打印工单、报价单、交付单时，未来可逐步展示：

- 本次施工项目
- 配件明细
- 工时费
- 完工检查结果
- 专业作业说明
- 技师签字

## 6. 资料库与数字化手册的关系

这两者不是互相替代，而是并行关系。

### 6.1 资料库的作用

资料库负责“保存和快速查找原始资料”，重点是：

- PDF 原文件存档
- 按车型归类
- 方便技师快速打开查阅

### 6.2 数字化手册的作用

数字化手册负责“把高价值内容结构化”，重点是：

- 不用每次翻整本 PDF
- 关键步骤和参数直接进系统
- 能在工单和维修项目中直接调用

### 6.3 两者最终关系

理想状态下，一个维修项目既能关联：

- 原始 PDF 资料
- 数字化作业条目
- 扭矩与技术数据

这样既保留原始依据，又提升实际操作效率。

## 7. 关于 PDF 自动识别的思考

本项目未来可以考虑“上传 PDF 后辅助解析”，但原则上不应追求不受控的全自动入库。

更稳妥的路线是：

1. 上传 PDF 到车型资料库  
2. 系统自动抽取文本或 OCR  
3. 识别候选内容：
   - 维修项目名称
   - 步骤
   - 扭矩
   - 油液
   - 配件名称
4. 生成待审核草稿  
5. 人工确认后再进入正式标准库

核心原则：

- 可追溯
- 可审核
- 可回滚
- 不直接污染正式数据

## 8. 当前优先级建议

在实施顺序上，建议按以下节奏推进。

### 第一阶段：经营闭环先稳定

重点完成：

- 客户
- 车辆
- 工单
- 报价
- 配件
- 交付
- 体检

这是保证系统可实际经营使用的基础。

### 第二阶段：车型标准库做厚

重点完成：

- 每个车型具备基础保养项目
- 标准项目挂工时、价格、配件
- 工单能直接复用标准项目

这是让门店从“靠人记忆”转向“按标准执行”的关键。

### 第三阶段：资料库和数字化手册持续沉淀

重点完成：

- 车型资料库完善
- PDF 手册归档
- 关键维修条目结构化
- 扭矩和检查点数据沉淀

这是形成长期竞争壁垒的关键阶段。

## 9. 最终产品目标

项目最终希望实现的，不只是“系统能用”，而是门店获得以下能力：

- 经营流程清晰可控
- 车辆档案长期积累
- 标准维修项目可直接调用
- 技师施工有标准依据
- 配件和资料与车型强关联
- 客户单据更专业、更有说服力
- 门店逐步形成自己的维修知识资产库

最终让系统成为门店的“业务中枢 + 技术知识中枢”，而不是单纯的开单软件。
