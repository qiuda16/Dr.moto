# 企业级成熟化路线图（2026-04-20）

Updated: 2026-04-20

## 1. 当前判断

从单店可用、准生产版继续往“大厂交付产品”推进，下一阶段重点不应该再是大量堆新功能，而是：

- 把已有功能做成稳定、可验证、可追溯、可配置
- 把产品心智做得更统一
- 把工程和数据底座做得更可靠
- 把 AI / OCR 从“可用”继续打磨成“可信”

当前项目的强项已经很明确：

- 门店主流程已成型
- 标准车型 / 标准配件 / 标准资料三库已建立
- OCR、资料审核、手册分段、标准库回写链路已经跑通第一代能力
- 数据库结构已经开始做强约束和巡检
- 自动化测试和运行脚本基础不错

当前项目距离“大厂交付感”的主要差距，集中在以下 6 条主线。

## 2. 最值得继续做的 6 条主线

## 2.1 产品层：把“功能集合”变成“统一产品”

### 当前短板

- 一些页面还是偏“大工作台”，块很多、信息密度高
- 同一类对象虽然已经统一口径，但部分文案和交互仍不够像最终交付态
- 缺少“按角色 / 按场景”的默认工作流精简

### 继续成熟化方向

- 工单页继续做“默认简洁、按需展开”
- 把资料审核页和车型标准页的审阅体验继续收细
- 把空状态、错误状态、无权限状态、加载状态统一
- 把列表页筛选项做成更像 SaaS 的“预设视图”

### 建议优先级

- P0：统一空状态 / 错误状态 / 加载状态
- P0：工单页缺项提示和快捷动作再减一层冗余
- P1：资料库加入“审核预设视图”
- P1：客户库和车型库列表加入固定常用筛选

## 2.2 配置层：把“写死逻辑”继续变成“系统设置”

### 当前短板

- 现在已经有门店设置底座，但覆盖范围还不够大
- 仍有不少默认值和文案散落在业务页面

### 继续成熟化方向

- 把常用保养套餐推荐口径接入设置
- 把文档页眉 / 页脚 / 门店签章文案接入设置
- 把默认工位、默认交车提示、默认售后提醒接入设置
- 把资料审核默认状态、审核备注模板接入设置

### 建议优先级

- P0：单据模板默认值进入设置
- P1：套餐推荐文案进入设置
- P1：资料审核模板文案进入设置

## 2.3 工程层：把“能跑”继续变成“可发布”

### 当前短板

- 代码和脚本已经很多，但发版前检查还分散在多个命令里
- 临时文件、一次性脚本、验证产物仍需要持续收口
- 构建通过不代表整个系统具备统一的发版前验收动作

### 继续成熟化方向

- 建立统一“发版前巡检”动作
- 持续收口临时文件和历史实验产物
- 为关键模块补发版前检查脚本

### 建议优先级

- P0：统一 pre-release audit 脚本
- P0：继续清理根目录临时产物
- P1：导出发版巡检结果报告

## 2.4 质量层：把“测试通过”继续变成“回归可信”

### 当前短板

- 后端自动化测试已经不错，但还主要在 API 层
- 前端缺少真正的主流程自动化验证
- 极限输入、连续运行、并发请求这三类风险还没系统化压测

### 继续成熟化方向

- 补一条员工端 E2E 主链路
- 补资料上传 -> OCR -> 审核 -> 回写的脚本化回归
- 做超长输入 / 大文件 / 并发请求的专项验证

### 建议优先级

- P0：发版前统一跑后端测试 + 前端构建 + DB 巡检
- P1：补员工端关键 E2E
- P1：补 OCR 主链路自动回归
- P2：并发与连续运行压测

## 2.5 数据层：把“结构正确”继续变成“数据治理成熟”

### 当前短板

- 数据库结构已经稳了不少，但历史数据质量仍有尾项
- 工单与标准车型桥接还存在历史空值
- 资料识别后的回写还需要更多“确认后再入库”的防线

### 继续成熟化方向

- 建立历史工单车型补全工具流
- 建立标准车型确认后的强门禁
- 继续提高 OCR 已确认结果进入标准规格 / 标准作业卡 / 标准配件的精度

### 建议优先级

- P0：历史工单车型补全工具
- P0：车型未确认前严格禁止标准库写入
- P1：规格与步骤的确认态进一步结构化

## 2.6 AI 层：把“可用”继续变成“可信”

### 当前短板

- OCR + LLM 联合抽取已经有价值，但仍需人工审核
- 章节、步骤、参数、配件之间的语义绑定还不够稳
- 还缺“关键参数可信来源”的统一展示和校验

### 继续成熟化方向

- 强化页码、章节、来源片段的统一展示
- 参数写回时增加“已确认来源页”标记
- 让标准项目与手册章节的绑定关系更显式
- 继续提高扭矩、油液、规格、配件型号抽取精度

### 建议优先级

- P0：关键参数都必须带来源页码
- P1：标准项目与分段章节建立显式绑定表
- P1：分段 PDF 与作业卡建立更强映射
- P2：再推进本地模型对结构化抽取的参与度

## 3. 我最建议你下一阶段重点做的 10 件事

按“最像大厂交付”的收益排序，我建议优先做：

1. 做统一发版前巡检脚本
2. 继续清理根目录和交付包里的临时文件
3. 把单据模板默认值接入设置中心
4. 补工单主流程 E2E 回归
5. 补资料 OCR 主链路回归
6. 继续加强历史工单车型补全工具
7. 统一全站空状态 / 错误状态 / 加载状态
8. 让资料审核和标准库写入门禁更严格
9. 让关键参数全部显示来源页码和确认状态
10. 给主数据中心做固定“预设视图”和审核工作流

## 4. 如果按阶段推进，建议这样分

### 阶段 A：交付感收口

- 统一页面状态
- 清理目录与临时产物
- 强化设置中心覆盖范围
- 发版前巡检脚本

### 阶段 B：质量与数据治理

- 工单主链路 E2E
- OCR 主链路回归
- 历史数据补全工具
- 标准库写入门禁

### 阶段 C：AI 可信度增强

- 参数来源统一展示
- 标准项目与章节显式绑定
- 结构化抽取精度继续提升

## 5. 结论

当前项目已经不是“缺功能”，而是进入了“做成熟”的阶段。

真正像大厂交付版本，关键不在于页面再炫一点，而在于：

- 规则统一
- 数据可信
- 配置可控
- 发布可验
- 出问题能追

这也是下一阶段最值得投入的方向。
