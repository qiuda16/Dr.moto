# DrMoto 一期后台系统建设蓝图

## 1. 建设范围

一期目标是让门店内部完成“接车到交付”的闭环操作，并为后续客户侧查询与多端接入保留稳定接口。

本阶段只聚焦内部后台，不把 AI、客户端、小程序作为阻塞项。

## 2. 分层职责

### 2.1 Odoo 业务内核
- 客户档案
- 车辆档案
- 工单主对象
- 工单施工/配件明细
- 报价主对象与版本
- 库存领用与回退
- 单据编号、状态流、审计基础字段

### 2.2 BFF 统一接口层
- 员工端统一 API 入口
- 登录鉴权与角色权限
- 幂等控制
- 审计日志
- 数据裁剪与对外接口口径统一
- 文件上传、附件引用、单据下载入口

### 2.3 Web Staff 内部后台
- 接车建档
- 工单列表与详情
- 报价录入与发布
- 配件领用
- 交付出单
- 基础主数据查询

### 2.4 文件与单据
- 报价单 PDF
- 工单打印单
- 领料单
- 交付/结算单
- 现场图片与附件

## 4. 核心数据对象

- `Customer`
- `Vehicle`
- `CustomerVehicle`
- `WorkOrder`
- `WorkOrderLine`
- `InspectionRecord`
- `Quote`
- `QuoteVersion`
- `Part`
- `StockMove`
- `MaterialIssue`
- `DeliveryOrder`
- `Attachment`
- `AuditLog`

## 5. 工单状态流

建议一期先固定为：

`draft -> confirmed -> diagnosing -> quoted -> in_progress -> ready -> done`

补充状态：

- `cancel`
- `on_hold`（二期可加，一期暂不必做）

约束原则：

- 只有已报价工单才能进入施工
- 只有施工完成工单才能进入待交付/完成
- 所有状态切换必须记审计日志

## 6. 报价版本流

报价需要独立于工单状态管理，建议：

`draft -> submitted -> published -> void`

规则：

- 同一工单允许多版报价
- 客户端只能看到 `published`
- 门店可追溯每次报价内容和发布时间

- 工单中心
- 报价中心
- 配件库存
- 客户档案
- 车辆档案

## 9. 一期 API 范围

建议按下列接口域推进：

- `/auth/*`
- `/admin/customers/*`
- `/admin/vehicles/*`
- `/admin/work-orders/*`
- `/admin/quotes/*`
- `/admin/inventory/*`
- `/admin/attachments/*`
- `/admin/documents/*`
- `/admin/audit/*`

现有 `/mp/*` 接口可继续保留作为兼容层，但中长期建议把后台接口收口到 `/admin/*`。

## 10. 编码与中文支持要求

后台必须默认支持中文姓名、车牌备注、故障描述、报价说明、附件名与 PDF 文本。

落地要求：

- Web 页面统一 UTF-8
- BFF 请求与响应统一 UTF-8
- PostgreSQL 连接显式使用 UTF-8
- Odoo / BFF 文本字段不做 ASCII 假设
- 附件对象存储路径使用 ASCII 安全路径，原始中文文件名单独保存
- PDF 生成时必须使用支持中文的字体文件

## 11. 实施顺序

### Sprint 1
- 补齐客户、车辆、工单模型
- 固化工单状态机
- 完成后台菜单与工单列表/详情
- 确保中文录入、保存、查询不报错

### Sprint 2
- 补齐报价版本模型
- 完成报价录入、审核、发布
- 生成基础报价单 PDF

### Sprint 3
- 补齐配件与库存闭环
- 完成领料/退料
- 补齐交付与结算单

### Sprint 4
- 权限矩阵
- 审计与日志
- 备份恢复
- 运维监控
