# 维修手册识别后提炼指南

更新日期：2026-04-25

## 目标

当维修手册、保养手册、扭矩表、线路图、故障诊断页进入系统后，不只是做 OCR 识别，还要把内容提炼成可检索、可落库、可复用的结构化知识。

这份文档说明：

- 要提炼哪些项目
- 每个项目怎么提炼
- 提炼后落到系统里的什么位置
- 哪些内容要保留人工复核

## 总体原则

1. 先保结构，再提细节。
2. 先做能稳定自动化的字段，再做容易噪声化的字段。
3. 所有高风险数值都要保留来源页码和原文。
4. 章节、步骤、规格、零件、注意事项要尽量保持和原手册一致。
5. OCR 不确定时，宁可留空，也不要强行猜测。

## 推荐提炼顺序

1. 车型适用范围
2. 文档基本信息
3. 目录和章节结构
4. 标准规格和技术参数
5. 维修 / 保养 / 拆装步骤
6. 零件、耗材和服务项目
7. 故障诊断和检查流程
8. 安全警告和注意事项
9. 图号、页码和来源追溯

## 需要提炼的项目

### 1. 文档基本信息

要提炼：

- 文档标题
- 文档类型
- 品牌
- 车型
- 年款范围
- 发动机 / 变速箱 / 平台信息
- 语言
- 页数
- 版本号 / 修订号

提炼方法：

- 从封面、扉页、页眉、页脚优先识别。
- 文档标题要保留原始命名，同时做标准化名称。
- 年款、车型、发动机型号可通过封面和适用范围页交叉校验。

落库位置：

- `summary_json`
- `normalized_manual`
- `manual_template`

### 2. 车型适用范围

要提炼：

- 品牌
- 车型名
- 车系
- 年款范围
- 排量
- 发动机型号
- 变速箱型号
- 适用市场 / 地区

提炼方法：

- 优先从 `Applicable Models`、`Vehicle Application`、`Fitment`、`Scope` 等段落提取。
- 不同页出现的车型名要做合并，不要重复建档。
- 识别到多个车型时，保留完整范围，不要只取第一个。

落库位置：

- `summary_json`
- 车型绑定表
- `vehicles`

### 3. 目录和章节结构

要提炼：

- 目录项
- 章节标题
- 小节标题
- 页码
- 章节层级关系

提炼方法：

- 用目录页的页码反推章节边界。
- 用标题层级、编号格式、字体大小和缩进识别章节结构。
- 目录和正文标题冲突时，以正文出现频次更高、结构更完整的一方为准。

落库位置：

- `toc_segments`
- `summary_json.sections`
- `vehicle_knowledge_segments`

### 4. 标准规格和技术参数

要提炼：

- 扭矩值
- 机油容量
- 冷却液容量
- 制动液容量
- 胎压
- 间隙
- 电压
- 电阻
- 标准值 / 极限值 / 公差
- 紧固顺序

提炼方法：

- 优先识别表格中的数值项。
- 用单位维度做校验，避免把单位换算表、符号表误识别为规格。
- 同一规格如果在不同页出现多个版本，保留来源页码并做冲突标记。
- 扭矩类参数要单独抽取，不能混在普通文本里。

落库位置：

- `summary_json.specs`
- `vehicle_catalog_specs`
- `procedure_steps.torque_spec`

### 5. 维修 / 保养 / 拆装步骤

要提炼：

- 作业名称
- 前置条件
- 标准步骤
- 检查步骤
- 安装步骤
- 拆卸步骤
- 复装步骤
- 收尾检查
- 所需工具
- 扭矩要求
- 安全注意事项

提炼方法：

- 按动作动词切分步骤，例如“拆下”“检查”“安装”“拧紧”“确认”。
- 每一步尽量保持一条完整动作，不要把多个动作塞到一句里。
- 步骤中出现扭矩、间隙、液量时，单独提升为结构化字段。
- 如果是分章节流程，先按章节聚合，再拆成细步。

落库位置：

- `summary_json.procedures`
- `procedures`
- `procedure_steps`

### 6. 零件、耗材和服务项目

要提炼：

- 零件名称
- 零件编号
- 规格型号
- 耗材名称
- 建议更换周期
- 服务项目名称
- 服务工时
- 对应维修方法
- 适用车型

提炼方法：

- 从零件表、材料表、保养项目表、维修项目表中抽取。
- 零件和耗材要和具体作业步骤关联，避免孤立存储。
- 服务项目要与“更换”“清洁”“检查”“调整”等动作对应。

落库位置：

- `summary_json.parts`
- `vehicle_service_template_items`
- 相关工单模板

### 7. 故障诊断和检查流程

要提炼：

- 故障现象
- 可能原因
- 检查顺序
- 排除方法
- 诊断结果分支
- 需要测量的参数
- 是否继续拆检

提炼方法：

- 识别“如果…则…”“检查…”“确认…”“可能原因…”这类诊断句式。
- 保持诊断链路顺序，不要把流程压缩成单段总结。
- 每个分支都要保留页码来源，方便回查原文。

落库位置：

- `summary_json.troubleshooting`
- `vehicle_knowledge_segments`

### 8. 安全警告和注意事项

要提炼：

- 警告
- 注意
- 预防措施
- 高压 / 高温 / 易燃风险
- 电瓶断开要求
- 支撑和固定要求
- 个人防护要求

提炼方法：

- 所有 `WARNING`、`CAUTION`、`NOTE` 类标识要单独提取。
- 安全类内容优先于普通步骤，必要时要在步骤前展示。
- 涉及人身风险的内容必须保留原文语气，不做弱化。

落库位置：

- `summary_json.safety_notes`
- `vehicle_knowledge_segments`

### 9. 图号、页码和来源追溯

要提炼：

- 来源页码
- 图号
- 表号
- 章节号
- 原文片段
- 识别置信度

提炼方法：

- 每个结构化项目都尽量带上原始来源页。
- 图表内容不要只存结果，要保留可以回溯的页码和标题。
- 当 OCR 结果有歧义时，用来源追溯作为人工复核入口。

落库位置：

- `summary_json.source_pages`
- `toc_segments`
- `vehicle_knowledge_segments`

## 识别后应该产出的结构

系统里建议至少形成下面几类结果：

1. `manual_template`
   - 手册的整体结构化模板
2. `normalized_manual`
   - 去噪、标准化后的手册摘要
3. `summary_json`
   - 适合程序消费的统一摘要
4. `toc_segments`
   - 目录 / 章节切分结果
5. `vehicle_catalog_specs`
   - 车型标准规格
6. `vehicle_service_template_items`
   - 服务项 / 保养项
7. `procedures` 和 `procedure_steps`
   - 维修方法和步骤
8. `vehicle_knowledge_segments`
   - 可检索的知识片段

## 推荐的抽取规则

### 先抽“结构”，再抽“内容”

先识别：

- 封面
- 目录
- 标题
- 章节层级
- 表格边界

再识别：

- 规格值
- 步骤内容
- 零件和耗材
- 注意事项

### 先抽“稳定字段”，再抽“高噪声字段”

优先抽：

- 文档标题
- 车型范围
- 章节标题
- 扭矩
- 容量
- 服务项目名称

后抽：

- 诊断分支
- 图表说明
- 复杂表格
- 批注类文本

### 先保“原文”，再做“标准化”

每个关键知识点最好同时保留：

- 原文片段
- 标准化结果
- 来源页码
- 是否人工确认

## 常见噪声和处理方式

1. 单位换算表被误识别成规格
   - 处理方式：加入白名单和低价值规则，单独过滤。
2. 表头、页眉、页脚重复出现
   - 处理方式：去重后再入库。
3. 目录页与正文页标题不一致
   - 处理方式：正文优先，目录仅作索引参考。
4. 扭矩数字混入普通段落
   - 处理方式：结构化提取后单独挂到步骤或规格里。
5. OCR 乱码
   - 处理方式：保留原始片段，同时标记待复核。

## 建议的人工复核点

以下内容建议人工复核后再正式入库：

- 车型适用范围
- 扭矩和关键安全参数
- 与拆装顺序相关的步骤
- 故障诊断分支
- 零件编号
- 服务项目和工时

## 对接到当前系统的理解

当前系统已经把手册识别后的结果拆成了几层：

- OCR 原始解析
- 标准化摘要
- 章节和片段切分
- 规格入库
- 服务项目入库
- 维修方法和步骤入库
- 知识片段入库

所以对一份新维修手册来说，正确的目标不是“把全文存起来”，而是把它拆成可以直接服务下面这些场景的结构：

- 查某个车型的标准规格
- 看某个作业的完整步骤
- 找某个保养项目对应的维修方法
- 按章节检索原始知识片段
- 回查某条参数的来源页

## 最终建议

如果后面继续增强提炼能力，优先补这四块：

1. 章节识别更稳
2. 扭矩和容量提取更准
3. 诊断流程分支更完整
4. 零件 / 服务项和步骤的关联更强

这样你的维修手册就不仅是“看得见”，而是能真正“用得上”。
